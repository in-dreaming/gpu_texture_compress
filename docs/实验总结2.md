## 二、深度反向沉淀（核心价值）

> 这一节是本次会话最有价值的部分。三个失败实验和它们的根本原因，对未来 ASTC 编码器优化方向给出了清晰的边界条件。

### 2.1 ASTC 4x4 多 partition：bit budget 数学不等式

**完整实现**：~340 行 HLSL，包括：
- 官方 `hash52` + `select_partition` 的 HLSL 端口（小心处理 int 算术右移）
- 全块 16 像素 partition mask 计算
- 每 subset PCA（mean + 协方差 + power iteration + min/max 投影）
- 每像素权重计算（基于自身 subset 端点）
- QUANT_5 weights + QUANT_32 endpoints 的 ISE 打包
- 候选 seed 搜索 ({0,1,2,4,8,16,27,35} 8 个) + 5% 误差 bias 守卫

**结果**：47.78 dB（与单 partition 完全相同）

**根本原因（最重要的格式级洞察）**：

ASTC 4x4 块只有 128 bits，2-partition 的 header 开销显著高于单 partition：

| 项目 | 1-partition | 2-partition | 差异 |
|------|------------|-------------|------|
| Block mode | 11 | 11 | 0 |
| Partition count | 2 | 2 | 0 |
| Partition seed | 0 | 10 | +10 |
| CEM | 4 | 6 (matched) | +2 |
| **Header** | **17** | **29** | **+12** |
| Endpoint values | 6 | 12 | +6 values |

剩余 99 bits（vs 单 partition 111 bits），需要塞下：
- 12 个端点值（vs 单 partition 6 个）
- 16 个 weight

**bit 预算饱和分析**：

| 配置 | weights | endpoints | 总 (bits) | 注 |
|------|---------|-----------|----------|------|
| 单 part Q12+Q256 | 58 | 48 | 17+58+48=**123** | **当前最优** |
| 2-part Q4+Q40 | 32 | 64.7 | 29+32+65=126 | endpoints quint 编码 |
| 2-part Q5+Q32 | 38 | 60 | 29+38+60=**127** | **本次实验所选** |
| 2-part Q8+Q16 | 48 | 48 | 29+48+48=125 | weights 8 levels |

**关键比较**（误差量化模型）：
- 单 partition：$\sigma_{ep}^2 \approx (\frac{1}{256})^2$，$\sigma_{w}^2 \approx (\frac{1}{22})^2$（QUANT_12 步长）
- 2-partition：$\sigma_{ep}^2 \approx (\frac{1}{32})^2 = 64 \times$ 单 part，$\sigma_{w}^2 \approx (\frac{1}{8})^2 = 8 \times$ 单 part

每像素端点量化误差为 64 倍。即使 partition 拆分能减少 subset 内方差，5-bit 端点的量化噪声主导误差。在自然图像（Khronos LDR-RGB）的所有候选 seed 上，2-partition 都被单 partition 击败。

**追加证明（8x8+ 同样不可行）**：

进一步分析发现 ASTC encode_core 的 weight 网格**固定为 4x4（16 weights）**，不随 block size 变化。解码器 bilinear 插值 16 weights 到任意 NxM 块。所以：

> ASTC 8x8 / 10x10 / 12x12 + 2-partition 的 bit 预算与 ASTC 4x4 + 2-partition **完全相同**：99 bits（QUANT_5 weights + QUANT_32 endpoints）。

既然 4x4 多 partition 已被单 partition 击败，更大 block 必然如此（除非用更大 weight grid，但那是另一个项目）。

**结论**：在当前 4x4 weight grid 框架下，多 partition 在**所有** ASTC block size 上都不是合适的优化方向。

### 2.2 ASTC LSQ 端点细化：被 weight 归一化破坏的不变量

**完整实现**：BC7 风格 2x2 normal-equations LSQ + 错误守卫，置于 PCA 后、encoding 前：
1. 计算当前权重（projection-based）和误差
2. 解 2x2 LSQ：`ep0 = (CX-BY)/det`、`ep1 = (AY-BX)/det`
3. 重新量化 + 重新计算误差
4. 仅当 `new_err < prev_err` 才提交

**结果**：ASTC_4x4 47.78 → 47.42 dB（**-0.37 dB 回退**）。错误守卫 SHOULD 阻止回退但仍发生。

**根本原因**：

ASTC 的 `calculate_normal_weights` 对每块的实际 min/max projection 重新归一化权重到 [0,1]：

```hlsl
for (i = 0; i < BLOCK_SIZE; ++i) {
    float w = dot(vec_k, texel - ep0);
    minw = min(w, minw);
    maxw = max(w, maxw);
    projw[i] = w;
}
float invlen = 1.0f / (maxw - minw);
projw[i] = (projw[i] - minw) * invlen;  // 归一化到 [0,1]
```

这意味着：

> **权重值与 (ep1 - ep0) 无关，只依赖 axis 方向**。存储的端点仅作为 decoder 的 lerp 边界。

LSQ 在连续空间求出的 `ep0_new`、`ep1_new` 偏离了"数据最值"位置。当 decoder 用 `lerp(stored_ep0, stored_ep1, weight)` 重建时：
- 原投影范围 [a_min, a_max] 映射到归一化 [0, 1]
- 但 stored endpoints 不再在这个 [a_min, a_max] 上
- decoder 的 lerp 在错误的"基准线"上，导致重建超出实际数据范围

错误守卫的"new_err < prev_err"基于我的近似模型，但 ASTC 实际解码用 stored_ep 而非 LSQ 求出的端点，所以模型与现实脱节。

**对比 BC7 为何成功**：

BC7 编码器**直接用 stored endpoints 计算权重**：
- 没有归一化步骤
- weight = `dot(pixel - ep0, ep1-ep0) / |ep1-ep0|^2`
- decoder 解码时也用同样的公式
- LSQ 移动 endpoints 同时改变权重和重建，误差是对应的

ASTC 因为有归一化，"端点 LSQ"模式不能不加思索地从 BC7 移植。

**沉淀**：

> 同一个优化模式在不同格式上有效性截然不同。判断标准是看：**优化变量是否真正影响 decoder 重建过程**。BC7 端点是真变量；ASTC 端点是 lerp 边界（= axis 方向 + 数据极值），LSQ 必须作用在 axis 方向上而非端点位置。

这直接指明了正确方向（§2.4）。

### 2.3 ASTC block mode search：当前框架的工程边界

**目标**：对每个块尝试多种 block mode（不同权重 grid、quant level），按误差选最优。

**分析得出不可行**：

#### 4x4 块的 bit 预算饱和

| 配置 | header | weights | endpoints | 总 |
|------|-------|---------|-----------|---|
| Q12 + Q256（当前）| 17 | 58 | 48 | **123** |
| Q16 + Q256 | 17 | 64 | 48 | 129 ❌ |
| Q20 + Q256 | 17 | 70 | 48 | 135 ❌ |
| Q24 + Q256 | 17 | 74 | 48 | 139 ❌ |
| Q16 + Q192 | 17 | 64 | 46 | **127** ✓ |
| Q16 + Q128 | 17 | 64 | 42 | **123** ✓ |
| Q20 + Q128 | 17 | 70 | 42 | **129** ❌ |
| Q24 + Q64 | 17 | 74 | 36 | **127** ✓ |

候选 (Q16 + Q192/Q128) 和 (Q24 + Q64) 在数学上可行，能提供更多 weight 精度。

**但实际工程不可行**：
1. `encode_color` 硬编码输出 0-255（QUANT_256）
2. 其他端点 quant 需要：
   - 量化 RGB 到目标范围（round 到 0..127 或 0..63）
   - **新的 endpoint scramble 表**：QUANT_192 / Q128 / Q64 对应 ASTC scramble，与 weight scramble 完全不同（要 port `color_uquant_to_scrambled_pquant_tables`）
   - 修改 `bise_endpoints` 的 ISE 范围参数
3. 对每个块尝试多种 block mode 还需要：
   - 多次跑量化与 ISE 打包流程
   - 计算每种结果的真实重建误差
   - 选最低误差返回 → SPV 体积膨胀（已观察到 multipartition 添加时 SPV 从 104KB 涨到 586KB，引发 GPU pipeline 创建超时）

#### 大块（6x6+）的更大 weight grid

| 块 | weight grid | weights bits | 是否可行 |
|----|-------------|--------------|---------|
| 6x6 | 4x4（16）current | Q12 = 58 | ✓ |
| 6x6 | 5x5（25） | Q12 ≈ 90 | ❌ |
| 6x6 | 6x6（36） | Q12 ≈ 130 | ❌ |
| 8x8 | 4x4（16）current | Q12 = 58 | ✓ |
| 8x8 | 5x5（25） | Q8 = 75 | ❌ |
| 8x8 | 5x5（25） | Q5 = 58 | ✓ but coarser |

5x5 weight grid 需要：
- 新的 decimation 表（pixel ↔ grid 的 bilinear 系数）
- `BLOCK_SIZE` 从 16 解耦（当前 X_GRIDS*Y_GRIDS == 16）
- 重构 `calculate_normal_weights` 的 sample_texel/idx_grids/wt_grids

工程量约 1-2 周（深度重构）。

**结论**：block mode search 在当前 encoder 框架下边际收益小、工程成本高。需要先重构 encode_core 才能重新评估。

### 2.4 PCA 轴搜索：唯一落地的优化（实证 PCA 已近顶）

**思路**（基于 §2.2 的反向沉淀）：既然 LSQ 不能改变端点位置，那就在 axis 方向上搜索。

**实现**：在 `encode_block` 中 PCA 之后插入候选轴评估：

```hlsl
if (QualityLevel >= 1) {
    float best_err = estimate_axis_error(texels, ep0_pca, ep1_pca, weight_range);
    [unroll] for (int c = 0; c < 3; c++) {
        float4 axis = (c==0) ? float4(1,0,0,0) : (c==1) ? float4(0,1,0,0) : float4(0,0,1,0);
        find_endpoints_for_axis(texels, axis, mean, cand_ep0, cand_ep1);
        float cand_err = estimate_axis_error(texels, cand_ep0, cand_ep1, weight_range);
        if (cand_err < best_err) {
            best_err = cand_err;
            ep0 = cand_ep0;
            ep1 = cand_ep1;
        }
    }
}
```

**误差估计模型**（与 ASTC 实际重建一致）：
1. 投影到候选轴
2. 计算该轴的 min/max projection（模拟 weight 归一化）
3. 量化为 weight_range 个等级
4. 解码用 stored endpoints lerp
5. 累加 RGB 平方差

**结果（Khronos LDR-RGB，QL=1，3 张图）**：

| 格式 | 优化前 | 优化后 | Δ |
|------|--------|--------|---|
| ASTC_4x4 | 47.7843 | 47.7869 | +0.0026 |
| ASTC_5x4 | 37.3483 | 37.3496 | +0.0013 |
| ASTC_5x5 | 34.9998 | 35.0005 | +0.0007 |
| ASTC_6x5 | 34.0515 | 34.0522 | +0.0007 |
| ASTC_6x6 | 33.4232 | 33.4238 | +0.0006 |
| ASTC_8x5 | 32.8043 | 32.8043 | 0.0000 |
| ASTC_8x6 | 32.3665 | 32.3671 | +0.0006 |
| ASTC_8x8 | 31.6380 | 31.6383 | +0.0003 |
| ASTC_10x5 | 31.8787 | 31.8789 | +0.0002 |
| ASTC_10x6 | 31.5536 | 31.5540 | +0.0004 |
| ASTC_10x8 | 30.9795 | 30.9799 | +0.0004 |
| ASTC_10x10 | 30.5964 | 30.5976 | +0.0012 |
| ASTC_12x10 | 29.8720 | 29.8735 | +0.0015 |
| ASTC_12x12 | 29.2890 | 29.2892 | +0.0002 |

**全 14 ASTC 格式微涨，BC 格式不动**（修改仅限 encode_core）。**零回归**。

**洞察**：

> 增益普遍 < 0.01 dB —— 这不是失败，而是**实证证实 PCA 主轴对自然图像已近乎最优**。

PCA 的最优性在数学上是清楚的：对 L2 误差，PCA 主轴最大化 `k^T C k`（沿轴方差），从而最小化垂直轴方差（off-axis variance）。RGB 对齐轴只在 PCA 8 次幂迭代未充分收敛的边角块上偶尔胜出（典型情况：单通道方差远超其他通道，PCA 没收敛到正解）。

**这次实验的真正价值在于"画下边界"**：
- 之前不知道 PCA 是不是最优
- 之后明确：PCA 已基本到顶，要再涨需要换思路

## 三、ASTC 编码器质量的格式级天花板分析

基于本次三个失败实验的 cross-cutting 结论：

### 3.1 当前实现的本质参数

```
单 partition + CEM 8 + 4x4 weight grid + Q12 weights + Q256 endpoints
```

这是当前 `encode_core` 的固定组合，对所有 ASTC block size（4x4 ~ 12x12）。

### 3.2 此组合的天花板原理

对一个 4x4 块（BLOCK_SIZE=16）：
- 端点存储：`6 × 8 bits = 48 bits`，量化误差 `~ (1/256)^2`，几乎可忽略
- 权重存储：`16 × ~3.6 bits = 58 bits`，QUANT_12 量化误差 `~ (1/22)^2`
- 误差主要来自：**off-axis variance**（PCA 主轴的最优值）

```
expected_error ≈ block_off_axis_variance + (weight_quant_step / 2)^2 * |ep1-ep0|^2
```

PCA 已最小化 off-axis variance；Q12 已是固定 bit budget 下最高的 weight 量化。所以 47.78 dB 是这个组合下的**理论近似上限**。

### 3.3 突破天花板的路径（按工程复杂度排序）

| 路径 | 期望增益 | 工程量 | 备注 |
|------|---------|-------|------|
| Endpoint quant 灵活化（Q128/Q192）| +0.3 ~ 1 dB | 中等（~3 天） | 移植 ASTC color scramble 表，改 encode_color，加 block mode search |
| 大块的更大 weight grid（5x5、6x6 等）| +1 ~ 3 dB | 大（~1 周） | 重构 encode_core BLOCK_SIZE，新 decimation 表 |
| Dual plane 模式 | +2 ~ 3 dB（仅 normal map） | 中等 | 一个通道用独立 weight 流 |
| CEM 模式扩展（Offset+Scale, Luminance）| +0.2 ~ 1 dB | 大 | 多种端点编码格式 |

每项都是 1-2 周级别的深度工程。

## 四、工程教训（cross-cutting）

### 4.1 多 mirror 同步的隐性成本

项目有三处 shader 镜像（`sdk/`、`shaders/`、`shaders_link/`），每次修改必须三处同步。

**本次会话踩坑**：BC7 Mode 1 完整实现期间，曾观察到 ASTC_4x4 测试输出始终是单 partition 数值（47.78 dB），**不论怎么改 multipartition 文件都不生效**。原因是 dispatch shader 实际使用的是 `shaders/compress/astc_4x4.hlsl`（没有 multipartition include），而我修改的 `sdk/shaders/compress/astc_4x4.hlsl` 有 include 但根本不被使用。

**沉淀**：以后修改 ASTC 任何文件，必须确认三处 mirror 的 **include 链** 一致，而不仅仅是修改的目标文件本身一致。

### 4.2 SPV 体积爆炸引发的 GPU pipeline 创建超时

引入完整 multipartition 后，DXC 编译的 SPV 从 ~104 KB 暴涨到 586 KB（**5.6 倍**）。原因是 `[unroll]` 在 8 个候选 seed 上展开了完整 PCA + weight 计算 + 打包流程。

**结果**：runner 在 "Starting Experiment" 后挂死（GPU pipeline 创建超时）。

**沉淀**：
- HLSL 中 `[unroll]` 候选数量要严格控制（< 8 复杂操作或 < 16 简单操作）
- 编译失败后即使简化代码到 stub，可能仍有缓存或验证状态残留
- SPV 体积是 GPU 运行时性能的早期预警信号

### 4.3 错误守卫的局限性

错误守卫（committing only if error decreased）是 BC1/BC6H/BC7/ASTC 通用模式。但**只有当模型的"误差函数"忠实反映真实重建误差时，守卫才有效**。

ASTC LSQ 实验中：
- 我的误差函数：用 LSQ 求出的连续 ep0/ep1 计算 `lerp(ep0, ep1, t_quant)`
- ASTC 实际重建：用 quantized stored_ep0/ep1（= 数据最值）计算 `lerp(stored_ep0, stored_ep1, t_quant)`
- 二者不一致，导致守卫"认为"改进的修改实际是回退

**沉淀**：错误守卫不是"安全护栏"，它的有效性取决于 inner-error model 的保真度。对结构性差异较大的格式（如 ASTC vs BC7），守卫不能盲目通用。

## 六、下一阶段建议

如果继续投入 ASTC 编码器深度优化，按 ROI 排序：

### 6.1 高 ROI（推荐）

1. **BC4/BC5 评估方法修复**：在 `eval_metrics.cpp` 解码 BC4 时复制 R 到 G/B（约 10 行 C++）。这不是"算法改进"但能让 BC4/BC5 PSNR 反映真实质量（当前 14 dB / 18 dB 严重低估，实际单通道 >40 dB）。

2. **BC7 完整 mode 5 / mode 7 / mode 0 实现**：BC7 当前只有 Mode 6 + Mode 1。Mode 5（dual plane RGB+A）和 Mode 0（3 partition）能进一步提升 +1-2 dB，但相对于已经 48.81 的水平，每 dB 越来越难。

### 6.2 中 ROI（值得深度投入）

3. **ASTC endpoint quant 灵活化**：移植 ASTC color scramble 表，使 `encode_color` 支持 Q128/Q192/Q64。可解锁更高 weight 精度（Q16/Q20）+ 中等端点精度组合。期望 +0.3 ~ 1 dB（仅自然图像）。

4. **ASTC dual plane 模式**：normal map 等数据可+2-3 dB。不影响 LDR RGB 测试。

### 6.3 低 ROI（除非有特定数据需求）

5. **大块 weight grid 重构**：5x5 / 6x6 grid 可让 6x6+ 块利用更多像素信息。期望 +1-3 dB。但需要 BLOCK_SIZE 解耦 + 新 decimation 表，约 1 周工作量。

6. **CEM 模式扩展**（Offset+Scale, Luminance）：对特定数据图案有效。
