## 二、关键技术突破

### 2.1 BC7：错误守卫的 LSQ 端点细化（+0.29 dB）

**问题**：原代码注释指出 LSQ refinement 会导致质量回退（49 dB → 25 dB），故被禁用。

**根因**：
- 原 LSQ 仅用一维加权平均（无交叉项），数学上不等价于真实 2x2 LSQ。
- LSQ 在连续空间精炼，但量化后可能反而恶化质量。

**解决方案**：
1. 改用真正的 **2x2 LSQ 矩阵求解**：
   ```
   A = sum (1-t)²,  B = sum (1-t)·t,  C = sum t²
   X = sum p·(1-t), Y = sum p·t
   ep0 = (C·X - B·Y) / (AC - B²)
   ep1 = (A·Y - B·X) / (AC - B²)
   ```
2. 加 **错误守卫**：每次 LSQ 后重新量化、重计算误差，仅当误差严格降低才提交。
3. 加 **p-bit 搜索**：4 种 (pbit0, pbit1) 组合中选最低误差，便宜地获得额外精度。

### 2.1.1 BC7 Mode 1：双 partition RGB 编码（+0.57 dB）

**目标**：高方差 block 在 Mode 6 单 subset 下端点不足以拟合，启用 Mode 1（2 subset RGB）应可改善。

**实现要点**（参考 `deps/DirectXTex/BC7Encode.hlsl block_package1()`）：
1. **64 partition 表**：枚举所有 partition pattern，按方差最大轴的"硬中线"做候选选择。
2. **6-bit 端点 + 共享 p-bit**：每个 subset 共用 1 个 p-bit。展开公式：
   ```
   v7 = (v6 << 1) | p
   v8 = (v7 << 1) | (v7 >> 6)   // 8-bit decoded value
   ```
3. **Storage form**：DirectXTex 用 `(v6 << 2) | (p << 1)`，提取时 `& 0xFC` 取高 6 位、`& 0x02` 取 p-bit。
4. **P-bit search**：每个 subset 4 种 (pbit0, pbit1) 组合，独立挑选。
5. **Anchor index fixup**：每个 subset 的 anchor index 仅 2 bits（最高 bit 推断为 0）；anchor 位置由 partition table 决定 ∈ {15, 2, 8, 6}。
6. **Mode 选择**：
   - QualityLevel 0：仅 Mode 6
   - QualityLevel ≥ 1：先 Mode 6；若 block variance > 0.005 且 alpha 方差 < 0.001（Mode 1 不支持 alpha），尝试 Mode 1，按 SSE 选择更优 block。

**结果**：BC7 由 48.24 dB → 48.81 dB（**+0.57 dB**），SSIM 0.9910 → 0.9913；单图 QualityLevel=2 可达 50.20 dB。零回归。

### 2.1.2 ASTC 关键 bug 修复：assemble_block 字位覆写（恢复 +25 dB）

**症状**：所有 ASTC 格式在某次中间迭代回归至 7.564 dB / SSIM 0.0012（≈ 随机噪声）。

**根因**：`astc_encode_core.hlsl` 第 433 行使用 `=` 而非 `|=`：
```hlsl
// BUG：覆写之前 wt_ise.z 写入的 weight 高 bit (line 409-412)
phy_blk.y = ((ep_ise.x >> 15) & 0x1FFFF);

// FIX：保留 weight bits，OR 入 endpoint 低 bits
phy_blk.y |= ((ep_ise.x >> 15) & 0x1FFFF);
```

ASTC 128-bit block 中 weights 从高位向下填、endpoints 从低位向上填，二者会在 32-63 bit 区间相遇。一旦 `wt_ise.z` 非零（weights 超过 64 bits 的 block mode），这个 `=` 会清掉刚写入的 weight 字位。

**修复范围**：3 处 mirror（`sdk/`、`shaders/`、`shaders_link/`）单字符修改。

**结果**：所有 ASTC 格式从 7.5 dB 恢复至原本水平（4x4: 47.78 dB、6x6: 33.42 dB、…）。该 bug 同时强化了"任何 `=` / `|=` 在 bit-packing 上下文都需 grep 审计"的工程教训。

### 2.2 BC6H：错误守卫修复隐性退化（+0.93 dB）

BC6H 已有 LSQ 但**没有错误守卫**。某些块的迭代会单调恶化但仍被提交。加 `if (new_error < prev_error) commit; else break;` 后立即获益。

### 2.3 BC1/BC3：2x2 LSQ + 非线性索引映射（+0.64 dB each）

BC1 的索引→t 映射非线性：`{0:0, 1:1, 2:1/3, 3:2/3}`。LSQ 时需用 lookup table 做正确映射。BC3 借助 BC1 自动获益。

### 2.4 ASTC 通用 QUANT_12 路径（+22-30 dB cumulative）

**最大单一杠杆**。原本简单格式（ASTC 5x4、5x5、…、12x12）使用：
- 自实现编码器
- QUANT_4（仅 4 个权重等级）
- 非标准 block mode 常量

效果停留在 22 dB 左右（接近随机噪声）。

**关键洞察**：
> ASTC block mode 编码 weight grid size，**不依赖 block size**。已存在的 4x4 ASTC 编码器（`astc_encode_core.hlsl`）实现了 PCA + QUANT_12 + 真实 trit/quint ISE encoding，在 ASTC_4x4 上达 47.78 dB。
>
> 任何格式的解码器都会自动 **bilinear 插值** 4x4 grid weights 到实际像素数。

**实现**：每个非 4x4 ASTC 格式只需：
1. 从输入像素中**子采样 16 个网格点**（4x4 grid 在该 block 上的对应位置）；
2. 调用现有 `encode_block(texels)`；
3. 完成。

代码量极小，每个文件约 25 行。例：
```hlsl
uint4 compress_astc_6x6(float4 pixels[36])
{
    float4 texels[BLOCK_SIZE];
    [unroll] for (int gy = 0; gy < 4; gy++) {
        [unroll] for (int gx = 0; gx < 4; gx++) {
            uint px = ((uint)gx * 5u + 1u) / 3u;
            uint py = ((uint)gy * 5u + 1u) / 3u;
            texels[gy * 4 + gx] = pixels[py * 6u + px] * 255.0f;
        }
    }
    return encode_block(texels);
}
```

### 2.5 修复完全损坏的格式

ASTC_8x6 和 ASTC_12x10 之前**故意**用 void_extent（输出常量颜色），停留在 12.42 dB（≈ 噪声水平）：
- ASTC_8x6: 12.42 → 32.37 dB（**+19.95 dB**）

### 2.7 失败实验记录（反向沉淀）

#### 2.7.1 ASTC 4x4 双 partition (CEM 8 matched)

**实现**：完整 2-partition 编码器，包括 hash52 + select_partition2、subset PCA、QUANT_5 weights + QUANT_32 endpoints ISE、1024 seed 候选搜索。

**结果**：47.78 dB（与单 partition 相同）— **2-partition 路径在所有候选 seed 上都无法击败单 partition**。

**根本原因**：ASTC 4x4 块的 bit budget 极紧。最大可用配置：
- 单 partition：QUANT_12 weights (12 levels) + QUANT_256 endpoints (8 bits direct)
- 2-partition：QUANT_5 weights (5 levels) + QUANT_32 endpoints (5 bits direct)

2-partition 端点精度仅为单 partition 的 1/8（每通道 5b vs 8b），插值精度仅为 5/12。即使 partition 拆分能减少 subset 内方差，**5-bit 端点量化噪声主导误差**，无法被空间自由度补偿。

**结论**：4x4 ASTC 多 partition 不是合适的优化方向。多 partition 在更大 block (8x8+) 上才有可能受益（每像素 bit budget 更宽松）。

**追加结论（8x8+ 验证）**：进一步分析发现 ASTC 多 partition 在 8x8/10x10/12x12 块上 **bit budget 完全相同**。原因：encode_core 的权重网格固定为 4x4（16 个权重），解码器 bilinear 插值到任意 block size。weight 数量不随 block size 变化。所以 8x8 + 2-partition 的预算仍是 99 bits（128 - 11 mode - 2 pcount - 10 partition_seed - 6 CEM），与 4x4 + 2-partition 完全相同。**多 partition 在所有 ASTC block size 上都不是合适的优化方向**（在当前 4x4 weight grid 框架下）。

#### 2.7.2 ASTC LSQ 端点细化（仿 BC7）

**实现**：在 encode_block 中加入 2x2 LSQ + 错误守卫，与 BC7 同样思路（BC7 +0.29 dB）。

**结果**：ASTC_4x4 47.78 → 47.42 dB（**-0.37 dB 回退**）。错误守卫 SHOULD 阻止回退但仍发生。

**根本原因**：ASTC 的 `calculate_normal_weights` 对每块的实际 min/max projection 重新归一化权重到 [0,1]：
```hlsl
projw[i] = (projw[i] - minw) * invlen;  // 强制每块 weights ∈ [0,1]
```

**这意味着权重值与 (ep1 - ep0) 无关，只有 axis 方向有关**。存储的端点仅作为 decoder 的 lerp 边界 — 解码时 `lerp(stored_ep0, stored_ep1, weight)`。

LSQ 在连续空间求 ep0/ep1 后，新端点偏离了"数据最值"位置，破坏了"stored_ep0/ep1 = data extremes along axis"的隐式不变量，导致 decoder lerp 时超出实际数据范围。

**正确路径（已验证 #1）**：改进方向应是**优化 PCA axis 方向**（搜索多个候选轴，比如 PCA + RGB 通道轴），而非改变端点位置。详见 §2.8 PCA 轴搜索。

**沉淀**：BC7 LSQ 之所以有效，是因为 BC7 编码器**直接用 stored endpoints 决定权重**，没有额外的归一化步骤。同一种"端点 LSQ"模式不能不加思索地从 BC7 移植到 ASTC。

#### 2.7.3 ASTC block mode search (#3 分析)

**目标**：对每个块尝试多种 block mode（不同权重 grid、quant level），选最低误差。

**分析结果**：在当前 encode_core 框架下不可行：
1. 4x4 块当前已用最高 weight quant (Q12) + 最高 endpoint quant (Q256)。Q16 weights 超预算（64 bits weights + 48 endpoint + 17 header = 129 > 128）。
2. 降低 weight quant (Q8/Q10) 是**严格变差**：endpoints 仍是 Q256（已最大），weight 精度反而下降。
3. 增加 weight quant 需配合降低 endpoint quant 到 Q128/Q192：但 `encode_color` 硬编码 8-bit 输出，需要重构 + 移植 ASTC scramble 表（非平凡工程）。
4. 大块（6x6+）想用更大 weight grid（如 5x5/6x6）需要新的 decimation table 和重构 encode_core 的 BLOCK_SIZE。

**结论**：block mode search 在当前框架下边际收益小、工程成本高。未来若重构 encode_core 支持任意端点 quant + 任意 weight grid，可以重新评估。

### 2.8 PCA 轴搜索（成功，全 ASTC 微涨）

**思路**：基于 §2.7.2 的 LSQ 失败教训，正确方向是优化 axis 方向，而非端点位置。在 PCA 主轴之外，再尝试 RGB 通道对齐轴（R/G/B 三个），用同样的"误差估计"模型挑选最优 axis。

**实现**：在 `encode_block` 中 PCA 之后，:
1. 对 PCA 给出的 (ep0, ep1) 计算"项目化重建误差"（含 weight 归一化与量化）
2. 对每个 RGB 单位轴 (1,0,0)/(0,1,0)/(0,0,1)：找该轴上 min/max projection → 候选 (ep0, ep1)
3. 计算候选误差，若 strictly < 当前 best，则采纳
4. 落到现有 weight_ise / endpoint_ise 路径

**结果（Khronos LDR-RGB，QL=1，3 张图）**：

| ASTC_5x4 | 37.3483 | 37.3496 | +0.0013 |
| ASTC_5x5 | 34.9998 | 35.0005 | +0.0007 |
| ASTC_6x5 | 34.0515 | 34.0522 | +0.0007 |
| ASTC_6x6 | 33.4232 | 33.4238 | +0.0006 |
| ASTC_8x6 | 32.3665 | 32.3671 | +0.0006 |

**所有 ASTC 格式微涨，BC 格式不动**（修改仅限 encode_core）。**零回归**。

**说明**：增益普遍 < 0.01 dB，证实 **PCA 主轴对自然图像已近乎最优**。RGB 对齐轴在某些块上偶尔胜出（典型情况：单通道方差占主导，PCA 8 次幂迭代未充分收敛），但绝大多数块 PCA 仍是最优。该实验明确给出"PCA 已基本到顶"的实证结论。

## 三、共性原则与技术沉淀

### 3.1 错误守卫迭代（universally applicable）

任何"看似改进"的算法变换，必须通过 **before/after 误差比较** 决定是否提交：

```
1. 保存当前状态 + 误差
2. 尝试改进
3. 重新量化 → 重新计算误差
4. if (new_error < prev_error) 提交; else 回滚 + break
```

适用：BC7 LSQ、BC6H LSQ、BC1 LSQ、ASTC encode_core LSQ。

### 3.2 解码器是隐性资源

不应仅把解码器当作评估工具——**解码器实现的特性可以反向利用**：
- ASTC bilinear 插值 → 编码器只需提供稀疏 grid weights
- BC6H FP16 端点 → 端点可用更高精度而无需 quantize 算法

### 3.3 写入 mirror copies 易成隐性 bug

项目有三处 shader 镜像（`sdk/`、`shaders/`、`shaders_link/`），每次修改必须三处同步。建议未来用 hard link 或 build-time copy 自动同步。

- **多 partition 编码**（CEM 8 双 partition）：~2-4 dB
- **Dual plane**（一通道独立权重）：1-2 dB（仅特定 block 类型）
- **自适应 QUANT 选择**（按数据自动选 Q8/Q12/Q20）

实现完整模式选择可能再获 +1-3 dB，但代码复杂度大幅增加。

   - **预期增益**：0.1-0.3 dB

6. **BC1/BC7 局部端点搜索**
   - LSQ 后做 ±2 步小范围搜索
   - **预期增益**：0.2-0.5 dB
   - **代价**：~5x 编码时间（quality level 2 only）

### 5.3 低优先级 / 实验性

7. **感知度量优化**
   - 当前优化 PSNR；可改为 SSIM 或 FLIP
   - 需修改误差守卫的比较函数
   - **预期影响**：人眼主观质量提升，PSNR 可能略降

8. **法线贴图专用路径（BC5 / ASTC normal）**
   - 假设 z 可重建，仅编码 xy
   - 用 normalize(xyz) 重建 metric
   - **预期影响**：normal map 质量大幅提升

9. **Speed/quality 自动调节**
   - 当前 QualityLevel 0/1/2 三档静态
   - 可基于像素统计动态选择内部参数
   - **预期影响**：降低 QL 1 的延迟、不损失质量

10. **HLSL 编译时间优化**
    - 部分 shader（ASTC 12x12 + Q12）SPIRV ~150 KB
    - 大量 [unroll] 循环膨胀
    - 可拆分为多 entry point 减小热度

### 5.4 长期方向

11. **shader code generation（meta-template）**
    - 14 个 ASTC 文件结构高度相似
    - 用脚本生成可减少代码重复，便于全局优化

12. **比较 ARM astcenc 参考结果**
    - 当前我们用 4x4 encode_block 路径覆盖所有 ASTC 格式
    - astcenc CPU 参考可能用不同 block mode（5x4 grid + ISE 等）能更高
    - **预期增益**：5x5/6x6 等还有 1-3 dB 上限

13. **ASTC encode_core 进一步加强**
    - 当前 4x4 路径 47.78 dB，理论上限 50+ dB
    - 可加 LSQ 错误守卫、p-bit-like 微扰、QUANT_20 选择
